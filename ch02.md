# 2. 느려진 서비스, 어디부터 봐야 할까

## DB 커넥션 풀
전체 응답 시간에서 DB를 연결하고 종료하는 시간은 꽤 많이 차지한다. 매번 연결하는게 아니라 커넥션을 미리 생성해서 보관하는 커넥션 풀을 이용하자

## 주요 요소
### 커넥션 풀 크기  
peak 시간대를고 려하여 최소 크기와 최대 크기를 지정하자. 
### 커넥션 대기 시간  
무작정 긴게 좋은게 아닌 것이, 만약에 지연 생겨서 클라이언트가 재요청 보내면 동시 트래픽이 더 증가하게 됨.  
기본 timeout = 30s 인 경우 많은데 위험하다. 제한된 커넥션 풀에 대기하느라 병목이 생김. 차라리 에러를 내보내서 동시 트래픽 & 대기 트래픽을 줄이는게 좋음. 
### 최대 유휴 시간 지정
DB가 끊어버리기 전에 커넥션 풀에서 제거해서 예상치 못한 동작 막자. 불안하면 유효성 검사를 요청 전에 하면 됨.
### 캐시를 도입하기
로컬 캐시는 빠르긴한데 배포 등으로 재시작하면 날라감 & 여러 서버면 쓰기 힘들다 --> 봐가면서 Redis 써라 ㅋㅋ
### 캐시 사전 적재
트래픽이 순간적으로 급증하는 패턴을 보인다면 미리 적재하는 것도 좋음
### 캐시 무효화
적당한 주기와 데이터 도메인 특성에 따라 적당한 무효화 주기 or 로직 설정 필요
### GC & Memory usage
파일들을 한꺼번에 내보내는 로직(즉, 메모리에 그대로 적재)이 많이 몰리면 메모리가 터질 수도 있음
- StreamingResponse으로 8KB 정도의 작은 단위로 끊어 흘려보내기. 메모리상 큰 파일 생성하지 말고 로컬에 생성한다음 스트림하는 방법. DB에서 데이터 받아와서 하는 동작도 마찬가지로 스트림 형태로 받는 것을 고려
### 트래픽
- Gzip으로 긴 텍스트 응답 압축 (근데 방화벽 지나갈 때 풀릴 수도 있으닌까 double check) -> 트래픽 비용 아끼자...
- 응답 헤더의 `Accept-Encoding`을 붙이면 처리할 수 있는 압축 알고리즘을 알릴 수 있다. 
- 브라우저 캐시를 이용하여 정적 자원의 재활용을 도모하자.
- CDN을 이용하여 컨텐츠 집약적인 요청을 효율적으로 분기하자.
- 파일 크기에 제한을 둬서 과도한 트래픽 증가를 막자
- 트래픽이 늘린다면 서버 scale은 조정이 쉽지만 DB는 어렵다(비용도 크다...) 대기 처리하여 수용 가능한 트래픽만 받자
