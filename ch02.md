# 2. 느려진 서비스, 어디부터 봐야 할까

## DB 커넥션 풀
전체 응답 시간에서 DB 연결/종료 시간이 꽤 큰 비중을 차지한다. 매번 연결하지 말고, 커넥션을 미리 생성해 보관하는 커넥션 풀을 이용하자.

## 주요 요소
### 커넥션 풀 크기
피크 시간대를 고려해 최소/최대 크기를 지정한다.

### 커넥션 대기 시간
무작정 길게 잡는 게 답은 아니다. 지연이 생기면 클라이언트 재요청이 늘어 동시 트래픽이 증가한다. 기본 timeout(예: 30s)은 오히려 위험할 수 있다. 제한된 커넥션 풀에서 대기 병목이 생기므로, 적절한 시점에 에러를 반환해 동시/대기 트래픽을 줄이는 편이 낫다.

### 최대 유휴 시간 지정
DB가 끊기기 전에 커넥션 풀에서 제거해 예상치 못한 동작을 막자. 불안하다면 요청 전에 유효성 검사를 수행한다.

### 캐시 도입
로컬 캐시는 빠르지만 배포/재시작 시 사라지고, 다중 서버 환경에서는 공유가 어렵다. 상황에 맞춰 Redis 같은 외부 캐시를 고려하자.

### 캐시 사전 적재
트래픽이 순간적으로 급증하는 패턴이 있다면 미리 적재하는 것도 효과적이다.

### 캐시 무효화
데이터 도메인 특성과 주기에 맞춰 적절한 무효화 주기/로직을 설계한다.

### GC & Memory usage
파일을 한꺼번에 메모리에 적재하는 로직이 몰리면 메모리가 터질 수 있다.
- `StreamingResponse`로 8KB 정도의 작은 단위로 끊어 흘려보내기
- 큰 파일은 메모리에 만들지 말고 로컬에 생성한 뒤 스트리밍
- DB에서 데이터를 받아오는 동작도 스트림 형태로 처리하는 것을 고려

### 트래픽
- Gzip으로 긴 텍스트 응답을 압축한다. 다만 방화벽을 통과하면서 풀리는 경우가 있으니 확인한다.
- 응답 헤더의 `Accept-Encoding`으로 처리 가능한 압축 알고리즘을 알린다.
- 브라우저 캐시를 이용해 정적 자원을 재활용한다.
- CDN을 이용해 콘텐츠 집약적인 요청을 효율적으로 분기한다.
- 파일 크기 제한을 두어 과도한 트래픽 증가를 막는다.
- 트래픽이 늘어나면 서버 스케일링은 비교적 쉽지만 DB는 어렵고 비용도 크다. 대기/거절 처리를 통해 수용 가능한 트래픽만 받도록 한다.
