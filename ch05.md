# 비동기 토픽
비동기에 대한 고민: 과연 오래 기다릴 필요가 있는가? 일부의 실패로 인해 전체가 실패되어야 하는가?

비동기 예시
- 푸시 서비스
- 포인트 지급
- 등록한 컨텐츠를 검색 엔진에도 등록
- 인증 메일 보내기

비동기의 조건
1. 연동에 약간의 시차가 생겨도 문제가 되지 않는다
2. 실패했을 때 재시도가 가능하다
3. 연동에 실패했을 때, 나중에 수동으로 처리가 가능하다
4. 연동 실패 시 무시해도 되는 기능이다

비동기 실행의 방법들...

## 별도 쓰레드로 실행하기
여기서 짚고 넘어갈 점이 있다.

### Python
비동기 vs 쓰레드 — 다르다.
- `asyncio`: 단일 쓰레드 기반 코루틴. `await` 시점에 이벤트 루프가 다른 코루틴을 실행. 동시성 모델일 뿐 추가 실행 리소스를 만들지는 않는다.
- `concurrent.futures.ThreadPoolExecutor`: OS 쓰레드 여러 개로 작업을 동시에 처리. I/O 대기 중 GIL을 반납하므로 I/O bound에선 실질적인 동시성을 얻지만, CPU bound는 GIL에 막혀 진짜 병렬 실행은 되지 않는다.
- `asyncio`와 `ProcessPoolExecutor`는 **층위가 다른 개념**이라 조합 가능.
- FastAPI는 non-blocking이지 병렬 처리가 아니다.


| 특징       | 스레드                      | 코루틴               |
| -------- | ------------------------ | ----------------- |
| 관리 주체    | 운영체제(*java의 경량 쓰레드는 JVM) | 개발자               |
| 멀티태스킹 방식 | 선점형 멀티태스킹                | 협력적 멀티태스킹         |
| 메모리      | 쓰레드마다 독립적인 스택 할당         | 스택 공유 또는 작은 공간 할당 |

### Java
Python(< 3.14t, free-threaded 빌드 이전)은 GIL로 막히지만, Java는 OS 쓰레드 수준의 진짜 병렬 실행이 가능하다.
```Java
@Service
public class EmailService {
    
    @Async  // 호출 즉시 반환, 별도 쓰레드에서 실행
    public CompletableFuture<Void> sendEmail(String to) {
        // 오래 걸리는 작업
        emailClient.send(to);
        return CompletableFuture.completedFuture(null);
    }
}
```
Thread-safe한 자료형으로 `ConcurrentHashMap`이 있다.

다시 돌아와서...
별도 쓰레드로 실행하는 경우, 호출 쓰레드로 Exception이 전파되지 않으니 유의해야 한다.

쓰레드 풀을 이용하는 이유: 쓰레드는 매번 새로 만들면 생성 비용이 들고, 각자 스택 메모리도 차지한다. 풀을 사용하면 미리 만들어 둔 쓰레드를 재사용하고, 풀 내부에서 자원을 관리할 수 있다.

## 메시징
생산자(게시자)가 메시징 시스템에 메시지를 쌓으면 소비자(구독자)가 그것을 받아 필요한 작업을 진행한다.

이점
- 시스템이 서로에게 영향을 주지 않는다. A의 트래픽이 증가해도 B에 그 영향이 바로 전이되지 않고, B의 성능이 저하되더라도 A는 영향받지 않는다.
- 용이한 확장성 -> worker만 추가하면 된다.


| | Kafka | RabbitMQ | Redis Pub/Sub |
|---|---|---|---|
| **모델** | Log-based (append-only) | Message Queue (broker) | Fire-and-forget |
| **중심 개념** | Consumer가 offset 관리 | Broker가 delivery 관리 | 채널에 뿌리고 끝 |
| **데이터 보존** | 디스크 보존 (기본 7일) | ACK 후 삭제 | 보존 없음 |
- Kafka: Pull 방식
- RabbitMQ: Push 방식. 브로커가 구독자에게 메시지를 전달하므로 브로커에서 병목이 발생할 수 있음
- Redis Pub/Sub: Broadcast + Forget

### 생산자 고려 사항
메시지 유실 시 대책은?
- 무시한다
- 재시도한다 — 다만 이 경우 중복 수신을 방지해야 한다. 고유 식별자를 부여하면 방지 가능
- 실패 로그를 남긴다

추가로, DB 트랜잭션이 끝난 뒤에 메시지를 발송해야 한다. 트랜잭션 중간에 발송하면 이후 롤백되더라도 메시지는 이미 나가버린 상태가 된다.

### 소비자 고려 사항
중복 처리가 발생할 수 있는 상황
- 생산자가 두 번 보냄 -> 고유 ID 부여로 해결. 저장은 DB 테이블이나 in-memory set으로 구현
- 소비 과정에서 오류가 나서 메시지를 재수신 -> 멱등성을 가지도록 API 설계

추가로, 큐가 가득 차지 않도록 모니터링도 필요하다.

### 메시지의 종류
- 이벤트: 보통 상태(데이터) 변경과 관련 있다. 꼭 그런 것만은 아니다 (예: 로그인 실패, 상품 정보 조회). 정해진 수신자가 없고, 발생한 사건에 관심 있는 소비자가 구독해서 받아 가는 방식이라 소비자 확장에 적합하다.
  - 예: "배송을 완료함" -> MQ -> 알림 서비스(알림 보내기), 주문 서비스(주문 상태를 완료로 변경)
- 커맨드: 특정 수신자에게 무언가를 요청함. 메시지를 받는 측의 기능 실행에 초점.


## 트랜잭션 아웃박스 패턴
메시지를 DB에 담아 전달한다.

과정
1. 비즈니스 로직 수행과 함께 필요한 DB 변경
2. 보낼 메시지를 아웃박스 테이블에 함께 기록 (같은 트랜잭션 안에서)
3. 별도의 메시지 중계 프로세스가 주기적으로 폴링해서 읽어옴 -> 메시징 시스템으로 전송

이점
1. 메시지 데이터가 유실되지 않음
2. 글로벌(분산) 트랜잭션 없이도 DB 트랜잭션 롤백 범위 안에 메시지 발행을 포함시킬 수 있음

기록 방식
- 발송 상태 컬럼 = 대기 | 완료 | 실패
- 성공적으로 전송한 마지막 메시지 ID를 별도로 기록 -> 다음번에는 그 이후 ID부터 처리

## 배치 처리
일정 간격으로 데이터를 모아서 전송한다.
예: 결제 승인 데이터를 모아 다음 날 보내기, 택배 발송 요청 데이터를 1시간 간격으로 보내기

방법들
- 정해진 시간과 규칙에 맞게 파일을 업로드
- 읽기 전용으로 DB를 열어주기

### 재처리 기능 만들기
지정한 시간에 전송하지 못한 경우를 위해 일정 시간 후 재전송하는 기능을 구현하자.
수동으로도 배치를 쉽게 실행할 수 있는 명령어/API를 달아 두면 편리하다.

tip) 전송할 데이터가 없다고 파일을 만들지 않으면 -> 받는 측 입장에서는 오늘 처리할 데이터가 없는 것인지, 오류로 업로드가 누락된 것인지 알 수 없다. 빈 파일이라도 만들자.

## Change Data Capture
DB의 데이터가 변경되면 그 내용을 외부에 통지하는 기능. 변경 사항은 레코드 단위로 전달된다.

활용 사례
1. 두 DB 간 데이터 연동이 필요한 경우 -> DB 사이에 CDC 처리기를 둔다.
2. 메시징 시스템에 변경분을 전파 -> 여러 시스템에 변경 데이터를 전달 가능

CDC 처리기는 어디까지 처리했는지(오프셋/LSN 등)를 기록해 두어야 한다. 기록이 없으면, 재시작 시 어디서부터 처리해야 할지 알 수 없어 다운타임 동안 발생한 변경분을 놓치거나 중복 처리하게 된다.

### CDC가 유용한 케이스
변경분 전파가 필요하지만 애플리케이션 코드를 손대기 어려울 때: 코드 변경 없이 DB 수준에서 외부 연동을 확장할 수 있어 편리하다.
