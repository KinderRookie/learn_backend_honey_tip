# 외부 연동이 문제일 때 살펴봐야 할 것들
## 우리는 문제가 없었는데
장애는 연동된 서비스에도 영향을 준다. 일례로, 대규모 트래픽이 감당 가능한 우리 서버가 인증 외부 연동 서비스를 이용하는 경우, 이 트래픽을 모두 외부 연동으로 보내면 장애가 발생할 수 있고, 그것이 역으로 우리 서비스에 영향을 끼칠 수 있다.

## 타임 아웃
연동 서비스에 타임아웃을 설정하지 않는 경우, 연동 서비스의 응답이 느릴 때 처리량이 급격히 떨어진다. 사용자가 새로고침 같은 방법으로 새로운 요청을 보내면 상황 악화
-> 타임아웃을 지정하여 사용자에게 에러를 응답하자. 동시 요청이 포화 상태가 되기 전에 전체 요청 수를 관리할 수 있다.

## 두 가지 타임 아웃
A서비스가 B 서비스로 연동을 하는 경우
```
A → B: 1.   연결 시도
A ← B: 1.1  연결 성공
A → B: 2.   요청 전송
A ← B: 3.   응답 전송
A → B: 4.   연결 종료
```
연결 타임아웃: 네트워크 상황 / 연동 대상 서버의 상태에 영향 받음
읽기 타임아웃: 요청을 전송한 뒤 응답을 받기까지 시간이 오래 걸림. 읽기 타임아웃은 주의해야 하는 것이, 연동 서비스는 정상 처리 중일 수도 있음

## 재시도
조건들
- 단순 조회 기능
- 연결 타임 아웃
- 멱등성을 가진 변경 기능
포인트 차감 서비스 같이 데이터 변경이 이뤄지는 경우는 사용에 조심해야 함
멱등성이란: 연산을 여러 번 적용해도 결과가 달라지지 않는 성질 ex: 게시글에 좋아요 누르기
새로 좋아요 -> 200
이미 좋아요 눌렀음 -> 변화 없음, 200

재시도 횟수 : 1-2회가 적당
간격: 지수적으로 확장해도 좋음
다만, 재시도 주의 사항: 재시도 폭풍 안티패턴 -> 느려진 외부 연동 서비스가 배로 요청을 받게 되면서 점점 느려짐

## 동시 요청 제한
벌크헤드 패턴: 각 구성 요소를 격리함으로써 한 구성 요소의 장애가 다른 구성 요소에 영향을 주지 않도록 하는 설계 패턴
ex
A서비스로 요청 300개 옴, 연동 서비스는 200개만 처리가능 -> 200개만 보내고 100개는 에러 처리 (혹은 Queue에 넣기)

## 서킷 브레이커
외부 연동 서비스가 지속적으로 장애가 발생하는 동안은 재시도를 해도 의미가 없음, 요청을 보내면 안됨(어차피 수초 이상 대기해야 함) -> 그냥 빠르게 에러 화면을 보는게 좋음. 이런 것들을 fail-fast 라고 한다.

서킷 브레이커를 도입
```
요청 → [닫힘] → 열기(실패 > 임계치) → [열림] → 반 열기(차단 시간 지남) → [반 열림]
[반 열림] → 닫기(성공) → [닫힘]
[반 열림] → 다시 열기(실패) → [열림]
```

임계치 -> 비율
시간 기준, ex: 10초 동안 오류 비율 50% 초과
개수 기준, ex: 100개 중 오류 비율 50% 초과

## 외부 연동과 DB 연동
### 외부 연동에 실패했을 때 트랜잭션을 롤백
트랜잭션 범위 안에 있는 경우, 롤백 시 DB에 변경한 데이터가 반영되지 않아 단순하다. 다만 읽기 타임아웃인 경우, 외부 연동 서비스에는 이미 데이터가 반영된 가능성을 고려해야 한다. 데이터 정합성을 체크해줘야 한다.
1. 주기적으로 두 데이터를 비교
2. 읽기 타임아웃이 발생한 경우, 이전 요청이 성공적으로 처리됐는지 확인하는 API를 호출
### 외부 연동 성공 했는데 DB 연동에 실패해서 롤백
취소 API 등을 호출해야 한다.

### 외부 연동이 느려질 때 DB 커넥션 풀 문제
다음과 같은 서비스를 가정해 보자.
1. DB 커넥션 풀에서 커넥션 가져옴
2. 0.1초 걸리는 DB 쿼리 실행
3. 4.8초 걸리는 외부 연동 실행
4. 0.1초 걸리는 DB 쿼리 실행
5. 커넥션 반납
이 경우, DB 쿼리를 실행하지 않아도 커넥션이 점유된 상태가 지속된다.
커넥션 풀이 제한적인 경우, 이 외부 연동 서비스 때문에 커넥션 풀이 고갈될 수 있다. 따라서 둘을 분리하는 방향을 생각해볼 수 있다. 다만 이렇게 하면 트랜잭션 범위에서 벗어나서 추가적인 조치가 필요하다.

추가로 HTTP도 커넥션 풀이 있고, 커넥션 유지 시간을 지정할 수 있다.

### 연동 서비스 이중화
쇼핑몰이 여러 결제 서비스를 두는 것처럼, 전체 서비스를 무너뜨리지 않도록 핵심이 되는 외부 연동 서비스는 여러 벤더를 이용하는 것이 좋다.
