# 동시성과 데이터
서로 다른 쓰레드가 하나의 데이터를 동시에 처리하기 시작하면 데이터가 꼬이기 시작한다.
특히 Singleton 객체의 필드 변수를 다중 쓰레드 환경에서 접근하는 경우 데이터가 꼬이기 쉽다 (race condition).

예시: read 후 write 사이에 끼어드는 경우
```plaintext
Thread A: READ  x  (x = 5를 읽기 시작)
Thread B:   WRITE x = 99
Thread A:        ... x로 연산 중 (근데 이미 99로 바뀜)
```

A가 읽은 값으로 **연산하는 도중** B가 덮어씀. A 입장에서 읽은 값이 **유효하지 않아진** 상황.

프로세스 수준 & DB 수준 모두 동시성 문제가 발생할 수 있다.

## 프로세스 수준 동시성 접근 제어
### lock을 이용한 제어
1. 공유 자원에 대해 잠금을 획득
2. 임계 영역(critical section, 공유 자원에 접근하는 코드) 실행
3. 잠금을 해제

예시
```java
import java.util.concurrent.locks.ReentrantLock;

public class Counter {
    private final ReentrantLock lock = new ReentrantLock();
    private int count = 0;

    public void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock(); // 반드시 finally에
        }
    }

    public int get() {
        lock.lock();
        try {
            return count;
        } finally {
            lock.unlock();
        }
    }
}
```

| | `synchronized` | `ReentrantLock` |
|--|--|--|
| tryLock | ❌ | ✅ |
| 타임아웃 | ❌ | ✅ |
| 인터럽트 | ❌ | ✅ |
| Fair 모드 | ❌ | ✅ |
| Condition | 1개 (wait/notify) | 여러 개 가능 |
| 코드 간결성 | ✅ | 복잡 |

단순한 임계구역이면 `synchronized`로 충분. **타임아웃, 인터럽트, 다중 Condition**이 필요할 때 `ReentrantLock` 쓰는 게 맞음.

### Semaphore
동시에 접근할 수 있는 쓰레드 수(permit)를 제한
1. 세마포어에서 퍼밋 획득 (allow_permit--)
2. 코드 실행
3. 퍼밋 반환 (allow_permit++)

### 읽기 쓰기 잠금
중간에 쓰기(put)가 끼어들지 않는다면 읽기(get)는 사실 동시에 수행되어도 된다.
일반 lock만 사용하면 읽기끼리도 직렬화되어 읽기 성능이 떨어진다. 읽기/쓰기 잠금을 이용하면 이 단점을 줄이면서 동시 접근 문제도 해결할 수 있다.
- 쓰기 잠금은 한 번에 한 쓰레드만 획득 가능
- 읽기 잠금은 여러 쓰레드가 동시에 획득 가능
- 어떤 쓰레드가 쓰기 잠금을 가지고 있으면 다른 쓰레드는 읽기 잠금을 획득할 수 없음
- 읽기 잠금이 모두 해제될 때까지 쓰기 잠금을 획득할 수 없음

```java
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class UserCache {
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
    private final ReentrantReadWriteLock.ReadLock readLock = lock.readLock();
    private final ReentrantReadWriteLock.WriteLock writeLock = lock.writeLock();

    private final Map<String, String> cache = new HashMap<>();

    // read → 여러 쓰레드 동시 진입 가능
    public String get(String key) {
        readLock.lock();
        try {
            return cache.get(key);
        } finally {
            readLock.unlock();
        }
    }

    // write → 혼자만 진입 가능 (read도 차단)
    public void put(String key, String value) {
        writeLock.lock();
        try {
            cache.put(key, value);
        } finally {
            writeLock.unlock();
        }
    }
}
```

## Atomic Type

`count++`는 사실 3단계임:

```
LOAD count
ADD 1
STORE count
```

이 3단계가 **쪼개지지 않고 하나의 연산처럼** 실행되는 걸 보장하는 타입.  
내부적으로 **CAS (Compare-And-Swap)** 라는 CPU 명령어를 사용.

```java
AtomicBoolean flag = new AtomicBoolean(false);
AtomicInteger count = new AtomicInteger(0);
AtomicLong bigCount = new AtomicLong(0L);
```

```java
AtomicInteger count = new AtomicInteger(0);

count.get();              // 읽기
count.set(10);            // 쓰기
count.incrementAndGet();  // ++count (증가 후 반환)
count.getAndIncrement();  // count++ (반환 후 증가)
count.addAndGet(5);       // += 5 후 반환
count.compareAndSet(5, 10); // 현재값이 5면 10으로 교체, 아니면 실패
```

### synchronized vs Atomic

```java
// synchronized 버전
private int count = 0;
public synchronized void increment() {
    count++;
}

// Atomic 버전
private AtomicInteger count = new AtomicInteger(0);
public void increment() {
    count.incrementAndGet(); // 락 없음
}
```

| | `synchronized` | `Atomic` |
|--|--|--|
| 방식 | JVM 모니터 락 (경합 시 OS mutex로 escalation) | CPU CAS 명령어 |
| 컨텍스트 스위칭 | 경합 시 발생 (블록/대기) | 없음 (실패 시 spin) |
| 경합 적을 때 | 비교적 느림 | 빠름 |
| 경합 많을 때 | 블록되어 CPU는 양보 | spin 반복으로 CPU 낭비 가능 |
| 복잡한 임계구역 | 가능 | 단일 변수 연산만 |

Atomic은 **단일 변수 연산**에서만 의미가 있다. 여러 변수를 묶어서 원자적으로 처리하려면 `ReentrantLock`이나 `synchronized`를 써야 한다.

### 동시성 지원 컬렉션
```java
Map<String, String> map = new HashMap<>();
Map<String, String> syncMap = Collections.synchronizedMap(map);
syncMap.put("key1", "val1"); // put 내부적으로 synchronized로 처리됨
```

or
```java
ConcurrentMap<String, String> map = new ConcurrentHashMap<>();
```
> `ConcurrentMap`은 인터페이스라 직접 인스턴스화할 수 없고, 구현체인 `ConcurrentHashMap` 등을 써야 한다. 버킷 단위로 잠금을 분할(lock striping)해서 `synchronizedMap`보다 동시성이 좋다.

or
```java
CopyOnWriteArrayList<Listener> listeners = new CopyOnWriteArrayList<>();
```
쓰기 시 내부 배열 복사본을 만들어 수정 후 원본을 교체하는 전략. 어떻게 보면 불변값을 사용해 동시성 문제를 해결한 셈. 읽기는 락 없이 가능하지만, 쓰기 비용이 크기 때문에 읽기 비중이 압도적으로 높을 때 적합하다.


## DB와 동시성
- 비관적 잠금: 먼저 접근한 트랜잭션이 잠금을 획득하고, 다른 트랜잭션은 잠금이 풀릴 때까지 대기한다. 충돌 가능성이 높거나 같은 데이터를 반드시 순차적으로 처리해야 할 때 사용한다.
- 낙관적 잠금: 명시적으로 잠금을 잡지 않고, 수정 시점에 버전 컬럼 값이 조회 시점과 같은지 확인한다. 값이 달라졌다면 다른 트랜잭션이 먼저 수정한 것이므로 적절한 오류 처리 후 트랜잭션을 롤백한다. 잠금 대기 시간이 없어서 빠르게 응답할 수 있다.

### 상황: splitQuery에서의 동시성 문제
```java
Subject subject = fromSqlRaw("select id, joinCount, ... from SUBJECT where id = ?", id);

joinToSubject(joinData, subject);

_ = fromSqlRaw("update SUBJECT set joinCount = ? where id = ?", subject.getJoinCount() + 1, subject.getId());
```

동시에 두 명이 이 코드를 실행하면 DB에서 발생하는 일
```plaintext
T1: SELECT joinCount → 10
T2: SELECT joinCount → 10
T1: UPDATE SET joinCount = 11
T2: UPDATE SET joinCount = 11   ← 12여야 함
```

`SELECT`로 읽은 값을 애플리케이션에서 증가시킨 뒤 `UPDATE`하면, 위처럼 나중에 실행된 `UPDATE`가 앞선 `UPDATE` 결과를 덮어쓸 수 있다. 이런 문제를 lost update라고 한다.

해결) 증분 쿼리 사용하기
`update SUBJECT set joinCount = joinCount + 1 where id = ?`
읽은 값을 다시 쓰지 말고, DB에서 바로 증가시키기

### 잠금 사용 후 항상 반환하기
try-finally 패턴을 사용한다.
```java
lock.lock();
try {
    // do something
} finally {
    lock.unlock();
}
```

### lock을 얻기 위해 대기하는 시간에 상한을 정해둘 것
ex) 5초간 락 획득 대기, 5초 초과 -> 에러
사용자에게 긴 대기 시간을 초래하는 것보다 낫다.

### 교착상태에 주의할 것
한 작업에서 두 개 이상의 자원 잠금을 획득하는 경우 deadlock에 빠지기 쉽다.
```plaintext
T1: A 잠금 획득    T2: B 잠금 획득
T1: B 잠금 대기    T2: A 잠금 대기
... 무한 대기!
```
해결 방법
1. 잠금 대기 시간을 설정
2. 여러 자원에 대한 잠금을 얻을 때는 항상 정해진 순서로 획득

## 단일 스레드 사용하기
사실 대부분의 동시성 문제는 여러 스레드가 하나의 자원에 동시에 접근해서 발생한다.
해결)
여러 작업 요청 스레드 -> 작업 큐 -> 단일 작업 스레드(상태 관리 스레드) -> 자원
Non-blocking / async I/O 사용 시 블로킹 연산을 최소화해야 함 -> 단일 스레드 방식이 적합하다.  

(이유: 단일 작업 스레드는 한 번에 하나의 작업만 처리하므로 공유 상태를 동시에 수정할 일이 줄어든다. 대신 중간에 블로킹 I/O가 있으면 큐 전체가 멈추기 때문에, 오래 걸리는 DB/파일/네트워크 작업은 비동기로 처리하거나 별도 워커에 위임해야 한다.)
