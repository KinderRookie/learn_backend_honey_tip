# 성능을 좌우하는 DB 설계와 쿼리

DB가 너무 느리다면 full scan이 돌고 있지 않은지 확인해야 한다. 인덱스를 타지 못해 전체 테이블을 훑는 경우가 많다. 모든 쿼리에 인덱스를 붙일 필요는 없지만, 자주 사용되는 쿼리가 있다면 그에 최적화된 인덱스를 설계해야 한다.

```sql
select id, category, writerId, title, content 
from article
where writerId = 10
order by id desc
limit 10
```
`writerId`에 인덱스가 없으면 full scan이 발생한다.

## 번외: `like`도 full scan을 유발
```sql
select id, category, writerId, title, content 
from article
where title like '%query%'
order by id desc
limit 10
```
`like`도 full scan을 유발하므로, 필요하다면 full-text search 기능을 고려한다.

## 단일 인덱스와 복합 인덱스
다음과 같은 쿼리가 있다고 하자.
```sql
select *
from activityLog
where userId = 123 and activityDate = '2024-01-01'
```
- 단일 인덱스(`userId`)만 있는 경우: 인덱스 컬럼을 빠르게 찾은 뒤 `activityDate`는 full scan으로 걸러낸다.
- 복합 인덱스(`userId`, `activityDate`)를 두는 경우: 하루에 데이터가 50만 건 이상 쌓일 정도라면 고려할 만하다.

## 선택도를 고려하기
예를 들어 `gender: M | F | U`처럼 값이 몇 개 안 되면 선택도가 낮다. 선택도가 높은 컬럼에 인덱스를 쓰면 조회 시간이 단축될 수 있지만, 항상 그런 것만은 아니다.
```sql
select *
from jobqueue
where status = 'W'
order by job_id
```
이처럼 특정 상태의 데이터를 빠르게 많이 읽어야 하는 경우에는 인덱스가 필요하다.

## 커버링 인덱스
인덱스에 필요한 모든 데이터가 포함된 인덱스를 말한다.

비교
```sql
select * from activityLog 
where activityDate = '2024-01-01' and activityType = 'login'
```
두 컬럼에 인덱스가 있다면 빠르게 찾을 수 있지만, 조회를 위해 각 레코드를 읽어 와야 한다.

```sql
select activityDate, activityType from activityLog 
where activityDate = '2024-01-01' and activityType = 'login'
```
이 경우 인덱스에 필요한 모든 데이터가 포함되어 있어 인덱스만으로 조회가 가능하다. 테이블 레코드를 읽는 과정이 줄어든다.

## 인덱스는 최소한만
```sql
create table reservation (
    id bigint primary key,
    name varchar(255) not null,
    reservationDate date not null,
    state char(2) not null,
    regDt time not null
); 
create index idx_reservation_name on reservation (reservationDate);
```

만약 이름으로 검색하고 싶다면

**하수:**
```sql
select * from reservation where name = 'John Doe';
```
이 경우 인덱스를 타지 못해 full scan을 돌린다.

**고수:** 요구 사항을 추가해 예약 일자를 조건에 포함한다.
```sql
select * from reservation 
where reservationDate = '2024-01-01' and name = 'John Doe';
```
이미 설정된 인덱스를 타므로 full scan 없이 이름 조회가 가능하다.

# 조회 성능 개선
## 미리 집계하기
- 설문 조사 항목 테이블
- 설문 조사 항목 - 유저 응답 내역 테이블
- 설문 조사 항목 - 유저 좋아요 내역 테이블

이렇게 있다고 하자. 각 항목(30개)을 뽑으면서 응답 유저 수, 좋아요 유저 수를 구하려면 다음과 같은 쿼리를 만들 수 있다.
```sql
select s.id, s.subject,
(select count(*) from answer a where a.id = s.id) as answerCount,
(select count(*) from like l where l.id = s.id) as likeCount
from survey s
```
목록 조회 1번 + 서브쿼리 30번 + 서브쿼리 30번 = 총 61번의 쿼리가 실행된다.

> 집계 데이터를 미리 계산해 별도 컬럼에 저장하자.

답변을 추가할 때, 좋아요를 누를 때 집계 컬럼을 갱신하면 count 조회 시 서브쿼리를 돌릴 필요가 없다.

## ⭐️ 페이지 기준 목록 조회 대신 ID 기준 목록 조회
데이터가 10만 개이고, 한 페이지에 10개만 보여준다고 하자. 총 페이지는 1만 개다.

처음 10개 읽는 쿼리
```sql
select id, name, writer
from article
order by id desc
limit 10 offset 0
```

마지막 1만 번째 페이지에 해당하는 데이터 읽기
```sql
select id, name, writer
from article
order by id desc
limit 10 offset 99990
```
실제로는 99,990개를 세고 난 다음 10개를 가져오는 셈이다.

```sql
select id, name, writer
from article
order by id desc
where deleted = false
limit 10 offset 99990
```
`deleted`에 인덱스가 없으면 실제 레코드를 읽어 와야 하므로 더 느려진다. 인덱스를 세워도 결국 세는 과정은 피하기 어렵다.

ID 기준으로 조회하면 카운트할 필요가 없다. `id desc` 기준이라고 생각해 보자.

```sql
select *
from article
order by id desc
limit 10
```
이렇게 첫 10개를 가져온다. 10번째 데이터의 `id`가 `9991`이라면 다음 10개는 이렇게 읽을 수 있다.

```sql
select *
from article
where id < 9991
order by id desc
limit 10
```
`id`는 인덱스를 타므로 9991보다 작은 값을 바로 찾을 수 있고, 카운트 과정도 필요 없다.

### 꿀팁
모바일 페이지는 페이지 이동보다 스크롤이 많다. 이때는 다음 데이터가 있는지 백엔드가 알려줘야 한다. `limit 11`로 한 개 더 조회해 판단할 수 있다.

```sql
select * from article
where id < 1001 and deleted = true
order by id desc
limit 11
```
11개가 조회되면 추가 데이터가 있다는 뜻이다. 응답은 10개만 제공하고, 추가 데이터 존재 여부를 `true`로 반환한다. 10개 이하라면 `false`로 반환한다.

## 조회 성능을 올리기
### 1. 조회 범위를 시간 기준으로 제한하기
뉴스 기사, 회원의 주문 내역 조회, 공지 사항 등 => 시간 범위를 제한하여 DB 조회 성능을 올릴 수 있다.
방법:
1. 특정 기간으로 한정하여 조회하기
   ```sql
   where order_dt >= '2025-08-09 00:00:00' and order_dt < '2025-08-10 00:00:00'
   ```
2. 최신의 데이터만 조회하기
   ```sql
   where check_month >= ?
   ```
이런 경우 날짜 컬럼에 인덱스를 달아주면 된다.
### 2. 전체 개수 세지 않기
```sql
select count(*)
from member 
where name like '%지은%'
```
count 쿼리는 데이터가 많아지는 경우 느려질 수 있다. 커버링 인덱스임에도 불구하고, 전체 인덱스를 스캔해야 한다.
### 3. 오래된 데이터 삭제 및 분리하기
- 데이터 개수를 일정하게 유지하기
- 데이터 증가 속도를 늦추기
	- ex) 과거의 데이터를 삭제하기 like 로그인 시도 내역

로그인 시도 내역 -> 이상치 탐지를 위한 목적이기 때문에, 오래 보관할 필요가 없다. 
180일 기준으로 오래된 내역은 삭제 or 180일 이내의 내역만 db에 담고 나머지는 별도의 보관소로 분리

**Tip)** `DELETE` 쿼리를 실행해도 실제 DB가 사용하는 디스크 용량은 일반적으로 줄어들지 않는다.
DELETE를 하면:
1. 해당 row를 **"삭제됨" 표시(dead tuple)** 만 함
2. 실제 데이터는 디스크에 그대로 남아있음
3. 진행 중인 트랜잭션이 이전 버전을 읽어야 할 수도 있으니까
따라서 데이터가 추가되고 변경되고 삭제 되는 과정에서 fragmentation이 발생한다. 실제는 최적화 작업을 진행해줘야 한다.
```sql
-- dead tuple 정리 (공간은 재사용 가능, OS 반환 X)
VACUUM table_name; 

-- OS에 실제 반환 (테이블 lock 걸림, 느림) 
VACUUM FULL table_name;
```
예외) 
`TRUNCATE`는 DELETE와 다르게, 내부적으로 페이지 자체를 통째로 deallocate하기 때문에 **즉시 공간 반환**.

### DB 확장하기
1. db를 수직 확장하기
2. db를 수평 확장하기
3. read replica db로 읽기 작업 전용을 두고, primary db에는 추가, 수정, 삭제 작업을 진행하고, 변경 사항을 read replica로 전파하기
4. 별도의 캐시 서버 도입하기

## 알아 두면 좋을 몇 가지 주의 사항
### 쿼리 타임아웃
시나리오: 사용자 동시 사용이 많아지면서 5초 걸리던 쿼리가 15초씩 걸린다. 사용자는 기대하는 시간이 넘으면 오류인 줄 알고 다시 요청을 보낸다. 이렇게 요청이 쌓이게 되고 재시도가 반복돼 동시에 처리해야 하는 요청 수가 기하급수적으로 늘어난다.
해결
1. 쿼리 타임아웃
   쿼리 실행 시간이 5초가 넘어가면 에러로 처리하고, DB 쿼리를 취소하고,  사용자에게 에러를 통보한다. 사용자는 에러 화면을 보게 되지만, 서버는 정상적으로 요청을 처리한 셈이 된다. 재시도를 하더라도, 이전 요청이 실행 중인 것이 아니라 처리가 되었으므로, 동시 요청 수 폭증을 막음
   **개선**: 쿼리 취소 + 큐
   ```
   5초 초과 → HTTP 에러 반환 + DB 쿼리 취소
           → 해당 요청을 백그라운드 큐에 enqueue
           → 완료되면 캐시에 저장
	사용자 재시도 → 캐시 히트 시 즉시 반환
   ```
   
2. (책에 없는 내용)요청 중복 제거 (Request Deduplication)
   ```
   사용자가 재시도를 해도 → 동일 요청이면 in-flight 요청 재사용

user_id + 요청 파라미터 → hash key
이미 처리 중이면 → 새 쿼리 실행 X, 기존 결과 대기
   ```

### 상태 변경 기능은 replica db에서 하지 않기
1. replica db는 순간적으로 primary db와 일치하지 않을 수 있음
2. 트랜잭션 문제가 발생할 수 있음. primary db에서 변경 트랜잭션이 완전히 안 끝났는데 그 중간에 replica db에서 읽으면 일관성이 깨질 수 있음.
따라서 회원 가입, 변경, 등록, 삭제와 같이 insert / update / delete 이용하는 경우 + 안에서 select를 해야 한다면 -> 주 db에서 select 하자.

### 배치 쿼리 실행 시간 증가
여러 데이터를 일괄로 조회하거나 집계하거나 생성하는 작업 -> group by로 묶고 sum, count등을 사용 or 특정 조건을 만족하는 데이터의 컬럼값을 일괄 변경 -> 한 번에 조회하는 데이터가 많아질 수록 느려진다.
해결
- 커버링 인덱스를 사용한다.
- 데이터를 잘게 쪼개어 진행한다. (ex: 한달 단위 -> 일일 단위 분해)
- insert...select 이런 애들을 select, insert 이렇게 분리하자.

### 타입이 다른 컬럼간 조인
user
- userId INT(10) *PK*
- name VARCHAR(10)

push
- id INT(10) *PK*
- receiverType VARCHAR(2)
- receiverId VARCHAR(200) *index*

```sql
select u.userId, u.name

```
-> 데이터 타입이 달라서 내부적으로 데이터 타입 캐스팅이 발생하고, 행마다 발생한다.  index를 활용하지 못한다. 
+ 문자열 타입을 비교할 때는 컬럼의 캐릭터셋이 같은지도 확인해야 한다. 

### 테이블 변경은 신중하게
mysql같은 경우 테이블 변경 시 새 테이블을 생성하고 원본 테이블을 복사한 뒤, 복사가 완료되면 새 테이블로 대체되는데, 이 과정에서 update, insert, delete 같은 DML 작업이 막힌다.

## 실패와 트랜잭션 고려하기
자주 발생하는 실수: 트랜잭션 없이 여러 db를 수정하는 행위

다음과 같은 사례
1. db 연결
2. 계약 데이터가 존재하는지 확인, 없다면 3, 4 실행
3. 계약 데이터 생성
4. 유저 상태 수정 (계약 완료)

이 과정에서 1~3은 잘 수행된 뒤, 4에서 에러가 발생하면, 계약 데이터는 이미 존재하지만 유저 상태는 미반영되어 유저는 다시 계약을 시도한다. 하지만 이미 계약 데이터가 존재하여 더 이상 계약을 수행할 수 없다. => 트랜잭션으로 롤백을 설정했어야 한다.

다만 모두 트랜잭션을 하는건 아니다.
1. 회원가입 form 받음
2. 유저 테이블에 생성
3. 가입 환영 이메일 보내기
여기서 3이 실패했다고, 회원가입을 모두 rollback 하는 건 아쉽다. 

```java
@Transactional
public void join(JoinRequest join) {
...
memberDAO.insert(member); // 유저 데이터 추가
try { 
mailclient.sendMail(...)
} catch (Exception ex) {
// 메일 발송 오류 무시
// 로그로 기록해 모니터링
}
}
```
외부 API 연동과 db 작업이 섞이면 트랜잭션 처리가 복잡해진다.
