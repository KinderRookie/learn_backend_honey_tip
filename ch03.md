# 성능을 좌우하는 DB 설계와 쿼리
DB가 너무 느리다? full-scan이 돌아가고 있지 않은지 체크할 필요가 있다. 인덱스를 못 타고 db를 다 훑고 있으니... 모두가 인덱스를 탈 필요 없지만, 자주 사용되는 쿼리가 있다면 그것에 최적화된 인덱스를 깔 필요가 있다.

```sql
select id, category, writerId, title, content 
from article
where writerId = 10
order by id desc
limit 10
```
여기서 `writerId`에 인덱스 없으면 full scan을 돌린다. 

***번외) 아 이건 몰랐다 ㅋㅋ***
```sql
select id, category, writerId, title, content 
from article
where title like '%query%'
order by id desc
limit 10
```
`like`도 full scan을 유발하니, full text search 기능을 사용해보자.

- 복합 인덱스(userId, activityDate)를 설정하는 경우:

## 선택도를 고려하기
gender: M | F | U 인 경우 선택도가 낮다는 뜻이다.  
선택도가 높은 컬럼에 인덱스를 사용하면 조회시간을 단축할 수 있다. 그러나 항상 그런 것만은 아니다. 
```sql
select *
from jobqueue
where status = 'W'
order by job_id
```
이런 경우 빠르게 많은 데이터 중 상태가 'W' 인 데이터를 가져와야 하기 때문에 인덱스 설정은 필요하다. 


## 커버링 인덱스
인덱스에 필요한 모든 데이터가 포함된 인덱스를 말한다. 
비교
```sql
select * from activityLog 
where activityDate = '2024-01-01' and activityType = 'login'
```
두 컬럼에 인덱스가 있다면 빠르게 찾을 수 있지만 조회를 위해 각 데이터를 읽어 온다. 

```sql
select activityDate, activityType from activityLog 
where activityDate = '2024-01-01' and activityType = 'login'
```
이런 경우 인덱스에 필요한 모든 데이터가 포함되어 있기 때문에 빠르게 찾을 수 있다. 데이터를 읽는 과정이 없어 빠르게 조회가 가능하다. 


## 인덱스는 최소한만
```sql
create table reservation (
    id bigint primary key,
    name varchar(255) not null,
    reservationDate date not null,
    state char(2) not null,
    regDt time not null
); 
create index idx_reservation_name on reservation (reservationDate);
```

만약에 이름으로 검색하고 싶다  
하수:
```sql
select * from reservation where name = 'John Doe';
```
이런 경우 인덱스를 타지 못하고 full scan을 돌린다. 

고수:  
요구 사항을 추가한다. 예약 일자를 조건에 추가한다.
```sql
select * from reservation 
where reservationDate = '2024-01-01' and name = 'John Doe';
```
미리 설정된 reservation 인덱스를 돌기 때문에 full scan 없이 예약자 이름으로 데이터 조회가 가능하다.

## 미리 집계하기
- 설문 조사 항목 테이블
- 설문 조사 항목 - 유저 응답 내역 테이블
- 설문 조사 항목 - 유저 좋아요 내역 테이블

이렇게 있다고 하자. 각 항목(30개)을 뽑으면서 각각에 대해 응답 유저 수, 좋아요 유저 수를 구하고 싶어 다음과 같은 쿼리를 짤 수도 있다:  
```sql
select s.id, s.subject,
(select count(*) from answer a where a.id = s.id) as answerCount,
(select count(*) from like l where l.id = s.id) as likeCount
from survey s
```
목록 조회 1번 + 서브 쿼리 30번 + 서브 쿼리 30번 = 총 61번의 쿼리 실행됨  
> 집계 데이터를 미리 계산해서 별도 컬럼에 담아 놓자!
답변을 추가할 때, 좋아요를 누를 때 집계 컬럼에 추가하면 count 조회 시 서브 쿼리를 돌릴 필요가 없다. 

## 페이지 기준 목록 조회 대신 ID 기준 목록 조회하기
데이터가 10만 개 있다고 하자. 한 페이지는 10개만 보여준다. 총 페이지는 1만 개이다. 
처음 10번째 데이터 읽는 쿼리
```sql
select id, name, writer
from article
order by id desc
limit 10 offset 0
```

마지막 1만 번째 페이지에 해당하는 데이터 읽기
```sql
select id, name, writer
from article
order by id desc
limit 10 offset 99990
```
이러면 실제로 데이터를 99,990개 세고 난 다음에 10개 데이터를 가져오는 것이다. 

```sql
select id, name, writer
from article
order by id desc
where deleted = false
limit 10 offset 99990
```
`deleted`에 인덱스라도 안 걸려있으면 실제로 데이터를 읽어와야 하기 때문에(왜? false/true 여부를 알기 위해 직접 가져와야 함) 더 암담해진다. 인덱스를 세워도 어차피 세야 하는 것은 마찬가지.  

ID 기준으로 세면 셀 필요가 없다. id를 desc 한다는 기준으로 생각해보면

```sql
select *
from article
order by id desc
limit 10
```
이러면 처음 10개의 데이터를 가져온다. 10번째 데이터의 id를 `9991`이라고 하자. 다음 10개 데이터는 이렇게 읽어올 수 있다:

```sql
select *
from article
where id < 9991
order by id desc
limit 10
```
id는 인덱스를 타므로 9991보다 작은 값을 바로 찾을 수 있다. 그리고 데이터를 셀 필요도 없다.   

### 꿀팁

모바일 페이지의 경우 페이지 이동보다는 스크롤 이동이 많은데 이 경우 다음 데이터가 존재하는지 백엔드가 알려줄 필요가 있다. 그럴 때에는 limit 11
```sql
select * from article
where id < 1001 and deleted = true
order by id desc
limit 11
```
이렇게 limit 11 결과 실제로 11개가 구해졌다면 읽을 데이터가 더 필요하다는 뜻이다. 응답으로 10개만 제공하고 추가 데이터 존재 유무를 true로 반환하면 된다. 10개 이하면 false로 반환한다.
