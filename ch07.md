# IO 병목을 해결하기
JVM의 경우, 가상 스레드를 이용하자.
플랫폼 스레드는 OS 스레드와 1:1로 매핑되며, 보통 CPU 코어 수만큼 생성된다. 이 플랫폼 스레드가 캐리어(carrier) 역할을 하면서 다수의 가상 스레드를 마운트해 가며 실행한다. 가상 스레드는 OS 수준 스레드보다 메모리를 덜 소비하고(예약 스택 크기가 작고) 컨텍스트 스위칭 비용도 작다.
IO 중심 -> 가상 스레드를 사용할 만하다. 블로킹 연산에 특히 효과적이다. 단, 플랫폼 스레드 수보다 충분히 많은 가상 스레드를 띄워야 이득이 생긴다. 개별 요청의 실행 속도가 빨라지는 게 아니라, 동시에 처리할 수 있는 처리량(throughput)이 늘어나는 것이다.
CPU 중심 -> CPU는 한정된 자원이기 때문에, 오히려 컨텍스트 스위칭 오버헤드로 인해 성능이 떨어질 수도 있다. 가상 스레드를 아무리 많이 만들어도 한 코어가 동시에 처리할 수 있는 연산은 하나뿐이니까.

## Non Blocking IO
먼저 blocking/non-blocking과 sync/async의 차이부터 정리하고 가야 한다.

호출 측에서 봤을 때 "기다리느냐 즉시 리턴하느냐"
- Non-blocking: 즉시 리턴 (준비 안 됐으면 `EAGAIN` 같은 신호로 알려줌)
- Blocking: 호출한 스레드가 대기 상태로 묶임

"커널 → 유저 버퍼 데이터 복사를 _누가_ 하느냐"
- async => 커널이 복사까지 끝낸 뒤 완료 통보만 해줌
- sync => 유저가 직접 `read()`를 호출해 가져옴

```
Phase 1: 데이터 준비 대기 (네트워크 패킷 도착, 디스크 읽기 등)
Phase 2: 커널 버퍼 → 유저 버퍼로 복사
```
- **Blocking vs Non-blocking** = Phase 1에서 기다리느냐 즉시 리턴하느냐
- **Sync vs Async** = Phase 2를 누가 하느냐

## **비유: 카페에서 커피 주문**

네 가지 케이스를 한 비유로 통일해서 보자.

**① Blocking + Sync (일반적인 Read/Write)** 주문하고 카운터 앞에서 멍하니 서서 기다림. 커피 나올 때까지 한 발짝도 못 움직임. 받으면 끝. → `read()` 호출 → 데이터가 올 때까지 스레드 정지 → 데이터 받음

**② Non-blocking + Sync (`O_NONBLOCK`)** 주문하고 자리에 앉음. 근데 30초마다 카운터에 가서 "내 거 됐어요?" 물어봐야 함. "아직요" 들으면 자리로 돌아옴. 됐으면 _본인이 직접_ 가서 받아옴. → `read()` 호출 → `EAGAIN` 즉시 리턴 → 잠깐 다른 일 → 또 `read()` → `EAGAIN` → … → 드디어 데이터 받음 → **이게 polling(busy-wait)이다.** 콜백이 아니다.

**③ I/O Multiplexing (`select`/`poll`/`epoll`)** 카페 여러 군데에 주문을 넣고 진동벨을 받음. 자리에 앉아서 진동벨이 울릴 때까지 기다림. 울리면 "3번 카페 됐어요" 알려줌. _본인이 3번 카페에 가서 받아옴._ → `select()` 호출 → 등록된 fd 중 하나라도 ready 될 때까지 blocking → ready 알림 → 본인이 `read()` 호출해서 데이터 가져옴

여기서 핵심: **진동벨은 "준비됐어"만 알려줌. 받는 건 본인 책임.** 그래서 실제로 받는 동안(`read()` 호출 구간)은 sync다.

**④ Async I/O (진짜 AIO)** 주문하면서 자리 번호를 알려줌. 직원이 _자리까지 커피를 가져다 줌_. 그 사이에 너는 책을 읽음. 도착하면 시그널/콜백으로 "여기 있어요" 통보. → `aio_read()` 호출 → 즉시 리턴 → 다른 일 함 → 커널이 데이터를 _유저 버퍼에 다 복사한 뒤_ 콜백/시그널로 통보 → **여기서 처음으로 콜백이 등장한다.**


## 서버 코드를 논블로킹 IO로 짠다
**이벤트 루프 + multiplexing** 모델
= 스레드를 많이 만들지 않고, **하나의 이벤트 루프가 epoll로 수많은 연결을 한꺼번에 다중화**해서 처리하도록 짠다.

```java
while (true) {
    client = accept() // 새 연결 받음
    spawn_thread(() => {
        data = read(client) // 데이터 올 때까지 이 스레드 정지
        process(data)
        write(client, response)
    })
}
```
- 연결 1만 개 = 스레드 1만 개. 각 스레드마다 메모리(~1MB 스택) + 컨텍스트 스위칭 오버헤드가 붙는다. 게다가 대부분의 스레드는 그냥 `read()`에서 멍 때리고 있는 중.

```java
epoll_fd = epoll_create()
epoll_add(epoll_fd, listen_sock)

while (true) {
    ready_fds = epoll_wait(epoll_fd)   // 여기서만 blocking
    for fd in ready_fds:
        if fd == listen_sock:
            client = accept()
            set_nonblocking(client)
            epoll_add(epoll_fd, client)
        else:
            data = read(fd)            // EAGAIN 안 남, 진짜 데이터 있음
            process_and_write(fd, data)
}
```
- 모든 소켓이 non-blocking → `read()`가 절대 블록되지 않음
- `epoll_wait`이 "데이터 온 fd 목록"을 알려줌
- 한 스레드로 1만 연결을 처리 가능 (각 연결은 대부분 idle하니까)
- CPU는 _실제로 일이 있을 때만_ 깨어남

### 리액터 패턴
리액터(이벤트 루프)가 이벤트를 대기하다가, 발생한 이벤트를 등록된 핸들러로 디스패치(전달)하는 과정을 반복한다.

특징
- **단일 스레드 기반** — 동시성을 스레드로 풀지 않고 이벤트 루프로 푼다
- **모든 핸들러는 non-blocking이어야 함** — 한 핸들러가 blocking call을 하면 이벤트 루프 전체가 멈춤
```plaintext
loop {
  1. epoll_wait()로 "지금 ready된 fd 목록" 받음     ← "계산 가능한 목록"
  2. 각 fd의 핸들러 콜백 실행                        ← "계산"
     - 핸들러는 read/write 하고 빠르게 리턴해야 함
     - 더 기다릴 게 있으면 fd를 epoll에 등록만 하고 즉시 리턴
  3. 다시 epoll_wait()로                             ← "대기"
     - 등록된 fd 중 하나라도 ready 되면 깨어남       ← "완료 알림"
}
```
